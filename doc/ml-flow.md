# ML 프로젝트 플로우 — 수신호 데이터 학습·예측

수신호 인식 모델을 어떤 데이터로, 어떻게 학습시키고, 실시간으로 어떻게 예측시키는지에 대한 전체 플로우.

## 1. 데이터

### 1-1. 주 데이터: 자체 수집 (확정)

수신호는 프로젝트마다 정의가 달라 표준 데이터셋이 없다. 배포 환경(사용자 웹캠, 정면 시점)과
동일한 조건에서 수집하는 자체 데이터가 소규모여도 가장 효과적이다 — Jester 실험(아래)으로 실증됨.

- 도구: `src/dataset/collect.py` (자동 연속 수집, 품질 필터, 손 이탈 경고 내장)
- 형식: `asset/<라벨>/<타임스탬프>.npy`, shape (30, 150) — 30프레임 × (양손 126 + 상체 포즈 24)
- 수량 가이드: 30개/라벨 = 검증용(~78%) · **100개/라벨 = 안정권(90%+) 목표** ·
  idle은 1.5~2배 (손 보이는/안 보이는 변형 모두 포함)
- **idle 클래스 필수** — 평상시 동작을 신호로 오인하는 것을 막는 클래스

### 1-2. 외부 데이터셋 조사와 Jester 실험 결과

조사했던 후보들 (연구 기록):

| 데이터셋 | 내용 | 다운로드 | 평가 |
|---|---|---|---|

| HaGRIDv2 | 손 제스처 이미지 100만 장, 제어용 제스처 포함 | [GitHub](https://github.com/hukenovs/hagrid) | 이미지 중심, 미사용 |
| SHREC'21 | 손 스켈레톤 시퀀스 180개 | [벤치마크](https://univr-vips.github.io/Shrec21/) | 형식 유사, 모델 벤치마크 참고용 |

| NATOPS 등 유도신호 계열 | 항공 마샬링 24종 등 | (연구용 신청) | 다중 사용자 일반화 단계에서 사전학습 후보 |
| AI Hub 수어/Ego-Vision | 한국수어 53.6만 클립 등 | [AI Hub](https://aihub.or.kr) (본인인증) | 어휘/시점 불일치로 제외 |

**Jester 실험 결론 (중요 교훈)**: `import_jester.py`로 4~5클래스 × 150개를 변환해 학습,
Jester 내부 test에서는 88~91%가 나왔지만 **실제 웹캠에서는 예측이 크게 빗나갔다.**
원인 = 도메인 갭 (동작 수행 방식·속도 차이) + 셀피 뷰 좌우 반전 (Jester는 비반전 영상이라
좌/우 신호가 반대로 학습됨). → **자체 수집 단독으로 전환.** 자체 데이터는 수집·추론이 같은
파이프라인을 통과하므로 두 문제가 구조적으로 없다. 외부 데이터는 라이선스 확인 후
사전학습 용도로만, 원본은 저장소에 커밋 금지.

## 2. 데이터 표현과 전처리

```
카메라 프레임 → MediaPipe HandLandmarker + PoseLandmarker (src/capture/extractor.py)
→ [왼손 63 | 오른손 63 | 상체 8관절 24] = 150차원/프레임 → 30프레임 시퀀스 = 샘플 (30, 150)
```

- **정규화**: 랜드마크는 이미 화면 기준 0~1. 손목 상대좌표화는 이동(모션) 정보를 없애므로 원시 좌표 유지
- **증강 = 특징 수준 도메인 랜덤화** (`src/training/augment.py`, 학습 배치에 실시간 적용):
  좌표 노이즈(센서 노이즈·저조도 떨림) / 손 프레임 드롭아웃(연기·가림·감지 실패) /
  시간 신축(동작 속도) / 이동·스케일(작업자 위치·거리) / 렌즈 왜곡(배럴·핀쿠션).
  부재 지점(0)은 변형하지 않으며 val/test에는 미적용. 끄기: `--no-augment`
- **분할**: train 70 / val 15 / test 15, 층화(stratified). 다중 사용자 데이터가 생기면 사람 단위 분할

## 3. 모델 선택

| 모델 | 특징 | 채택 |
|---|---|---|
| LSTM (2층, hidden 64) | 시퀀스 표준 베이스라인, 소량 데이터 가능, CPU 수 분 | ✅ 현재 사용 |
| 1D CNN | 빠르고 단순 | 비교 실험 후보 |
| Transformer | 대량 데이터에서 유리 | 데이터 확장 후 검토 |
| ST-GCN | 관절 그래프 모델링, 스켈레톤 SOTA 계열 | 고도화 단계 검토 |

원칙: **가장 단순한 모델로 시작해 성능이 부족할 때만 복잡도를 올린다.**
클래스 수는 `asset/` 폴더에서 동적으로 결정 (하드코딩 금지).

## 4. 학습 프로세스 (`src/training/train.py`)

```
asset/ 스캔 → 라벨 인코딩 → 층화 분할 → LSTM 학습 (cross-entropy, Adam)
→ validation 정확도 기준 early stopping (30 epoch 무개선 시 중단, 최고 모델 유지)
→ test 평가: 정확도 + 혼동 행렬 + 클래스별 정밀도/재현율
→ 저장: models/sign_classifier.pt (가중치 + 라벨 목록 + 설정 일체)
```

평가에서 볼 것:
- 혼동 행렬 — 어떤 신호끼리 헷갈리는지. 혼동 쌍은 동작 재정의 또는 통합 고려
  (실례: '멈춰'와 '오지마'가 혼동 쌍이어서 stop으로 통합했음)
- train↑ val↓ 이면 과적합 → 데이터 추가·증강·모델 축소
- **안전 관점: stop 재현율이 최우선 지표** (놓치면 사고). 오탐이 "정지 방향"이면 허용

## 5. 실시간 예측 플로우 (`src/inference/predict.py`)

```
웹캠 프레임 → 특징 추출 → 최근 30프레임 버퍼 (슬라이딩 윈도우) → 매 프레임 추론
→ 안전 가드: 윈도우의 포즈 감지율 30% 미만(사람 없음) → 모델 무시, "unknown"
→ 안정화 필터: 신뢰도 ≥ 0.8 + 연속 5회 동일 예측일 때만 확정
→ 확정 신호 전달 (publisher.py): 변화 시 즉시 + 1초 하트비트
→ 미달/미학습 동작 = "unknown" → 기계 쪽 안전 기본값(정지)
```

안정화 필터가 없으면 동작 중간 프레임마다 예측이 튄다.
UI: 확정 신호 + 클래스별 확률 막대 패널 + 임계값선 + 손 이탈 경고.

## 6. 반복 개선 루프 (현재 단계)

```
실시간 테스트에서 오분류 관찰 → 해당 라벨/상황 데이터 추가 수집 → 재학습 → 재평가
```

- 같은 라벨로 수집 태스크를 재실행하면 이어서 쌓인다 (덮어쓰기 아님)
- 수집 시 다양성(위치·속도·크기) > 개수. "테스트 때 하는 방식"이 수집분에 있어야 한다
- 모델 구조 변경보다 데이터 보강이 먼저다
