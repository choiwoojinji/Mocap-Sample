# ML 프로젝트 플로우 — 수신호 데이터 학습·예측

수신호 인식 모델을 어떤 데이터로, 어떻게 학습시키고, 실시간으로 어떻게 예측시킬지에 대한 전체 플로우.

## 1. 데이터셋 조사

### 1-1. 주 데이터: 자체 수집 (확정)

수신호는 프로젝트마다 정의가 달라 표준 데이터셋이 없다. 배포 환경(사용자 웹캠, 정면 시점)과
동일한 조건에서 수집하는 자체 데이터가 소규모여도 가장 효과적이다.

- 도구: `src/dataset/collect.py` (자동 연속 수집, 품질 필터 내장)
- 형식: `asset/<라벨>/<타임스탬프>.npy`, shape (30, 126) — 30프레임 × 두 손 랜드마크
- 권장: 라벨당 30개 이상 + **"신호없음(idle)" 클래스 필수** (평상시 동작 오인 방지)

### 1-2. 외부 데이터셋 조사 결과 (증강·참고용)

**결론**: 자체 수집으로 시작 → 일반화(다른 사람·환경)가 필요해지면 Jester/HaGRIDv2 영상에
우리 MediaPipe 추출기를 돌려 같은 (30, 126) 형식으로 변환해 증강한다.
외부 데이터는 라이선스 확인 후 사용하고 저장소에 원본을 커밋하지 않는다.

## 2. 데이터 표현과 전처리

```
원본 영상 프레임 → MediaPipe HandLandmarker → 두 손 × 21포인트 × (x,y,z) = 126차원/프레임
→ 30프레임 시퀀스 = 샘플 1개 (30, 126)
```

- **정규화**: 랜드마크는 이미 화면 크기 기준 0~1. 추가 정규화(손목 기준 상대좌표)는
  이동 동작 정보를 없앨 수 있으므로 기본은 원시 좌표 유지, 실험으로 검증
- **증강(augmentation)**: 좌표 노이즈 추가, 시간 축 이동/신축, 좌우 반전(손 슬롯 교환) —
  데이터가 적을 때 과적합 방지
- **분할**: train 70 / validation 15 / test 15 (%), 클래스 비율 유지(stratified).
  여러 사람 데이터가 생기면 사람 단위로 분할

## 3. 모델 선택

| 모델 | 특징 | 채택 |
|---|---|---|
| LSTM/GRU (1~2층) | 시퀀스 표준 베이스라인, 데이터 적어도 학습 가능, CPU로 수 분 | ✅ 베이스라인 |
| 1D CNN | 빠르고 단순, 짧은 시퀀스에 강함 | 비교 실험 후보 |
| Transformer | 데이터 많을 때 유리, 소규모에선 과적합 위험 | 데이터 확장 후 검토 |
| ST-GCN (그래프) | 관절 구조를 그래프로 모델링, 스켈레톤 인식 SOTA 계열 | 고도화 단계 검토 |

원칙: **가장 단순한 모델(LSTM)로 시작해 성능이 부족할 때만 복잡도를 올린다.**
클래스 수는 `asset/` 폴더에서 동적으로 결정 (하드코딩 금지, doc/design.md 참고).

## 4. 학습 프로세스 (`src/training/`)

```
asset/ 스캔 → 라벨 인코딩 → train/val/test 분할
→ (증강) → LSTM 학습 (cross-entropy, Adam)
→ epoch마다 validation 정확도 확인, 개선 없으면 조기 종료 (early stopping)
→ test 셋으로 최종 평가: 정확도 + 혼동 행렬 + 클래스별 정밀도/재현율
→ 산출물 저장: models/sign_classifier.*  +  라벨 목록(labels.json)
```

평가에서 볼 것:
- 혼동 행렬 — 어떤 신호끼리 헷갈리는지 (예: '오지마'와 '멈춰')
- train↑ val↓ 이면 과적합 → 데이터 추가·증강·모델 축소
- **안전 관점**: '멈춰'의 재현율(놓치지 않는 비율)이 가장 중요한 지표

## 5. 실시간 예측 플로우 (`src/inference/`)

```
웹캠 프레임 → 랜드마크 추출 → 최근 30프레임 버퍼 (슬라이딩 윈도우)
→ 매 프레임(또는 N프레임마다) 모델 추론 → 클래스 확률
→ 안정화 필터: 신뢰도 임계값(예: 0.8) + 연속 K회 동일 예측일 때만 확정
→ 확정된 신호 → 외부 전달 (기계 시뮬레이터)
→ 임계값 미달/미학습 동작 = "인식 불가" → 안전 기본값(정지) 유지
```

안정화 필터가 없으면 손을 움직이는 중간 동작마다 예측이 튄다.
"연속 K프레임 동일 + 신뢰도 임계값"이 오작동 억제의 핵심이다.

## 6. 반복 개선 루프

```
실시간 테스트에서 오분류 관찰 → 해당 상황 데이터 추가 수집 → 재학습 → 재평가
```

파이프라인 완성 후에는 이 루프가 작업의 본체가 된다.
모델 구조 변경보다 데이터 보강이 먼저다.
